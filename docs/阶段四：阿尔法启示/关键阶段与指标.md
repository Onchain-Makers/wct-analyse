# 阶段四 · 运作模型：自扩展链上监控（从公布地址 → α 告警）

> 这是阶段四的方法论核心：把 WCT 三部曲沉淀的"隐性手感"显性化成一套**可执行、可复用**的工作流。
> 配套：信号定义见 [阿尔法启示.md](阿尔法启示.md)（S1~S8）。本篇讲**怎么把信号跑起来**。

---

## 0. 一句话流程

> **公布地址 播种 → 每个新节点 [定性 → 剪枝 → 存快照 → 挂监听] → 事件驱动递归扩图（Mint→金库→子金库→…） → 在"活实体图 + 前瞻日历"上跑信号告警。**

---

## 1. 心法：公布地址 = 封面，真图在一跳之外

项目方官网/白皮书公布的，几乎都是**"运营合约"**（Token、Timelock、Config、质押、空投…）。
**真正装钱、真正出货的地址（金库、子金库、vesting、直发 EOA、做市簇）一个都不会公布**——但它们全部能从公布地址**一跳挖出来**。
所以第一动作不是研究这串地址本身，而是用它们**炸出隐藏的资金/控制地图**。

> WCT 实证：官网只给了 Token `0xeF4461891DfB3AC8572cCf7C794664A8DD927945` + 两个 Timelock + Config + Pauser + StakeWeight + RewardDistributor + Airdrop；而创世金库 `0xa86ca428512d0a18828898d2e656e9eb1b6ba6e7`、83 个子金库、3 个 MerkleVester、做市簇、40M 巨鲸——**全是挖出来的**。

## 2. T0 第一现场勘查（拿到公布地址，第一时间尽可能多采集）

目标：**信息储备**（多多益善，留作后用）+ **建立观察坐标**（解禁日、内部人、解锁日历）。

### 2.1 即时读（view 函数，秒出，先全存下来）
- Token：`totalSupply` / `decimals` / `name` / `symbol`
- 是否代理 + **实现地址** + **ProxyAdmin**（升级控制权）
- ⭐ **`transferRestrictionsDisabledAfter` = 解禁时间戳**（整个事件最重要的日历锚点）
- AccessControl 角色持有者（`DEFAULT_ADMIN_ROLE` / `MANAGER_ROLE` / `PAUSER` …）→ 控制权在谁手里
- Timelock：`getMinDelay`、proposer/executor 角色成员
- Config：读出它存的全套系统合约地址（**自动扩图**）
- 质押/分发/Vester：reward rate、总锁仓、merkle root、vesting schedule

### 2.2 一跳挖（事件，最关键的几件）
| 动作 | 拿到什么 | α 价值 |
|---|---|---|
| ⭐⭐ **Mint 事件**（from=0x0） | 铸给谁 = **创世金库**（没公布）；何时 = TGE | 翻开封面，顺藤摸全网 |
| ⭐⭐ **`SetAllowedFrom` / `SetAllowedTo`** | 锁仓期被预授权可转账的地址 = **项目方钦定的内部/分发钱包名单** | TGE 当天就拿到"内部人通讯录" |
| ⭐ Timelock **`CallScheduled`**（已排队未执行） | 未来要执行的升级/改参/转账，**在执行前可见** | 延迟期 = 预告窗口 |
| 角色授予 `RoleGranted/Revoked` | 控制权变更历史 | 治理/权限演化 |

> 口诀：**Mint 找金库，白名单找内部人，解禁戳定日历，Timelock 看预告。**

## 3. 自扩展实体图模型（核心引擎）

把上面挖到的一切，组织成一张**会自己生长的实体图**：节点 = 地址，边 = 资金流/控制关系。

### 3.1 节点 Schema（每个地址存的字段）
| 字段 | 说明 |
|---|---|
| `address` | 地址 |
| `type` | 合约 / EOA / Safe / CEX / vester / bridge / MM / unknown |
| `labels` | 角色标签（创世金库、子金库、白名单、直发EOA、做市簇…） |
| `first_seen` | 首次出现时间 + 来源（哪个节点的哪笔事件引出的） |
| `balance` | 当前持仓（定期刷新） |
| `insider_score` | 内部度/接近资金程度评分（决定监控优先级） |
| `watch` | 是否在持续监听（剪枝决定） |
| `snapshot` | 静态快照（owner/门槛/角色/代码/首笔 gas 来源…） |

### 3.2 定性（新节点入图先自动打标签）
监听到新地址 → 先判定它是什么，再决定怎么对待。判定来源：`is_contract`、`safe.safes_all`、`cex.addresses`、合约名/ABI、是否在白名单、收到的金额占比。

### 3.3 剪枝规则（展开 vs 截断）—— **防止图爆炸的关键**
| 新节点类型 | 处理 |
|---|---|
| 合约 / Safe 多签 | **展开 + 持续监控**（金库、vester、分发器） |
| 被 `allowedFrom/To` 白名单 | **重点展开**（钦定内部人） |
| 收到大额（占供应 %）的新鲜 EOA | **展开 + 重点盯**（直发份额=未来卖方） |
| 被已监控节点资助 gas 的新地址 | **展开**（疑似同操盘方，如做市簇舰队） |
| **CEX 地址** | **只标记、不展开**（否则顺币安爬到全世界） |
| 普通小额散户 EOA | 标记，不监控（噪音） |

> 我们手动做阶段一时，一直在凭直觉跑这套剪枝（碰 CEX 就停、碰 Safe 就深挖）；这里把它显性化成规则。

### 3.4 两层数据 + 关键状态重读
- **静态快照**：发现时读一次（身份、角色、owner/门槛、gas 来源…）。
- **动态流**：之后每一笔转账/事件，永久监听。
- ⚠️ **状态会变**：门槛、白名单、解禁时间戳、角色会改 → 关键合约**定期重读** + 靠事件（`SetAllowedFrom`/`ChangedThreshold`/`RoleGranted`）捕捉变更。

### 3.5 递归扩图（事件驱动）—— WCT 实例链路
```
公布: Token 0xeF44…7945
  └─[Mint 事件]→ 创世金库 0xa86ca428…（入图,展开,挂监听）
       └─[outflow]→ 子金库 cc9792 / 51f6 / c401（Safe→展开）
            └─ cc9792 →[outflow]→ MerkleVester 0x2ff1…（合约→展开）
                 └─→ 解锁路由 0x37badde9…→ 受益人 ~83（大额→盯）
            └─ c401 →[解禁前大额]→ 128 个新鲜 EOA（含 40M 巨鲸 0xd9fdb623…，展开重点盯）
       └─[gas 资助]→ 做市簇 hub 0x9a9c4219… + 舰队（同操盘方→展开）
  每个新节点都→[定性→剪枝→快照→监听]，循环。
```

## 4. 前瞻日历层（图之外的"时间维度"）

把所有"未来会发生的供给/控制事件"汇成一张前瞻表，和图叠加：
- **解禁时间戳**（`transferRestrictionsDisabledAfter`）= 弹药何时解锁
- **vesting schedule**（各 vester 的 calendar/interval）= 未来每个供给冲击点（如 WCT 2025-11 cliff 暴释 4,814 万）
- **Timelock 排队操作**（`CallScheduled` 的 eta）= 未来高权限动作的预告
- **质押/奖励释放节奏**

## 5. 信号 = 跑在"活图 + 日历"上的查询（接 [阿尔法启示.md](阿尔法启示.md) S1~S8）

图建好后，α 信号就是几条告警查询：
| 信号 | 在图上的查询 |
|---|---|
| **S2 发弹药** | 任一 [金库/vester 节点] → [新鲜 EOA] 的大额转账，且临近解禁日 |
| **S3 备货上架** | 任一 [内部节点] → [CEX 标记节点] 的净流入 spike（剔除内部搬运） |
| **S4 做市激活** | 出现 [新 funded 的 EOA 簇] 做双向高频 DEX 交易 |
| **S5 刷量** | 主池 per-address 买卖两边 net≈0 占比高 |
| **S6 顶部派发** | [内部节点→CEX] 净充值在价格抛物线顶端爆增（deposit/price 背离） |
| **S8 持续供给** | 前瞻日历里临近的解锁/cliff 节点 |

## 6. 诚实边界
- **CEX 黑箱**：充值可见、成交不可见 → 信号是"出货准备"的代理，非成交本身；实时性受限。
- **剪枝有取舍**：截断 CEX/散户会漏掉"散户里混的聪明钱"，但不剪枝图会爆炸——这是工程权衡。
- **博弈进化**：信号公开后操盘方会换更隐蔽路径（如经混币/新合约中转），α 会衰减，模型需迭代。
- **样本偏差**：本模型源自 WCT 单一标本，普适性/胜率需多代币回测验证（见 阿尔法启示.md 方向 2）。
